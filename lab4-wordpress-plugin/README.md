## 🎯 Цель и задачи работы

Целью данной лабораторной работы было освоить расширяемую модель данных WordPress и на практике реализовать собственный учебный плагин, включающий:

## 📝 Краткое описание выполненной работы

В ходе выполнения лабораторной работы я использовал уже существующую локальную установку WordPress, которая была ранее развёрнута в Docker в рамках предыдущих лабораторных работ. Это позволило мне не настраивать WordPress повторно, а сосредоточиться именно на разработке плагина и тестировании его функциональности.

Я создал учебный плагин USM Notes, который добавляет в WordPress новый раздел «Заметки». Для этого плагина я реализовал собственный Custom Post Type, пользовательскую таксономию приоритетов, метабокс для хранения даты напоминания, безопасное сохранение метаполя с помощью nonce, проверку обязательности даты и валидацию даты на недопустимость значений в прошлом.

Кроме того, я реализовал шорткод [usm_notes], который позволяет выводить заметки на страницах сайта, а также поддерживает фильтрацию по приоритету и дате. После завершения разработки я создал тестовые заметки, проверил работу шорткодов на отдельных страницах WordPress и подготовил плагин к загрузке в Git-репозиторий.

## 🧠 Общая идея реализации

В данной лабораторной работе я разработал собственный учебный плагин USM Notes. Плагин был создан в папке wp-content/plugins/usm-notes и подключался к уже существующему WordPress-сайту, запущенному в Docker.

Логика плагина строилась вокруг собственной сущности «Заметки», которая была реализована как отдельный Custom Post Type. Для классификации заметок по важности использовалась пользовательская таксономия Priority, включающая значения High, Medium и Low. Для хранения даты напоминания использовалось отдельное метаполе \_usm_due_date, которое редактировалось через специальный метабокс в административной панели.

На фронтенде вывод заметок был реализован через шорткод, который позволял:

- отображать все заметки;
- фильтровать заметки по приоритету;
- фильтровать заметки по дате напоминания;
- выводить сообщение, если подходящих заметок не найдено.

### 9. Сохранение и валидация даты напоминания

После отображения поля даты я реализовал сохранение значения метаполя через хук save_post.

В процессе сохранения были добавлены следующие проверки:

- проверка наличия и корректности nonce;
- защита от автосохранения;
- проверка прав пользователя на редактирование записи;
- проверка обязательности поля даты;
- валидация даты: дата напоминания не должна быть в прошлом.

Если поле не было заполнено, либо была указана дата в прошлом, плагин не сохранял значение и выводил сообщение об ошибке в административной панели.

Если дата была указана корректно, значение сохранялось в метаполе \_usm_due_date и отображалось при повторном редактировании записи.

### 12. Создание шорткода для отображения заметок

Следующим этапом я реализовал шорткод:

```text
[usm_notes]
```

а также его расширенные варианты:

```text
[usm_notes priority="high"]
[usm_notes before_date="2026-04-30"]
```

Для этого в плагине была добавлена функция обработки шорткода с помощью add_shortcode().

Шорткод поддерживал:

- отображение всех заметок;
- фильтрацию по priority;
- фильтрацию по before_date;
- вывод сообщения «Нет заметок с заданными параметрами», если записи по условиям не найдены.

Для выборки заметок использовался WP_Query с tax_query и meta_query в зависимости от переданных атрибутов шорткода.

- проверка даты и защита сохранения через nonce;
- отображение даты в списке заметок;
- шорткод для вывода заметок на страницах сайта.

### Реализованные примеры использования

- [usm_notes] — вывод всех заметок;
- [usm_notes priority="high"] — вывод заметок с высоким приоритетом;
- [usm_notes before_date="2026-04-30"] — вывод заметок с датой напоминания до указанной даты.

## 🚀 Инструкции по запуску проекта

Если WordPress уже развёрнут в Docker, для запуска проекта достаточно:

1. Запустить Docker-контейнеры WordPress, MySQL и phpMyAdmin.
2. Убедиться, что WordPress доступен по локальному адресу.
3. Поместить папку usm-notes в директорию:

```text
wp-content/plugins/
```

5. Перейти в раздел Plugins.
6. Найти плагин USM Notes и активировать его.

| 1   | Чем пользовательская таксономия принципиально отличается от метаполя? Приведи пример, когда выбрать таксономию, а когда — метаданные. | Пользовательская таксономия используется для классификации и группировки записей. Она подходит, когда одно и то же значение должно переиспользоваться у многих записей и участвовать в фильтрации, навигации и архивах. Например, приоритеты High / Medium / Low лучше хранить как таксономию. Метаполе используется для хранения конкретного дополнительного значения у одной записи. | 2   | Зачем нужен nonce при сохранении метаполей и что произойдёт, если его не проверять?                                                   | Nonce нужен для защиты формы от несанкционированной отправки и атак типа CSRF. | 3   | Какие аргументы register_post_type() и register_taxonomy() чаще всего важны для фронтенда и UX?                                       | Для фронтенда и UX особенно важны: public — определяет, будет ли сущность доступна пользователю и на сайте; has_archive — позволяет формировать архивные страницы и делать контент более доступным; labels — отвечают за понятные подписи в интерфейсе администратора; supports — задаёт, какие элементы доступны в редакторе (заголовок, редактор, автор, миниатюра); hierarchical — влияет на поведение таксономии и удобство группировки; show_in_rest — важно для совместимости с современным редактором Gutenberg и REST API. |
